# Участие в разработке Candy

Спасибо за интерес к Candy. Этот файл описывает требования к изменениям,
проверкам и сгенерированному коду.

## Подготовка окружения

Для основного модуля нужны:

- Go `1.25.6` или новее;
- Make;
- компилятор C для `go-sqlite3`;
- RPL `0.7.2`, для работы со сложными моделями;

```sh
git clone https://github.com/CandyCrafts/candy.git
cd candy
go mod download
make test
make build
```

## Оформление коммитов

Candy использует формат Conventional Commits:

```text
<type>(<scope>)!: <description>
```

`scope` и `!` необязательны. Заголовок, тело и footer пишутся на английском
языке, чтобы история репозитория оставалась единообразной.

```text
feat(parser): support multiple method returns
fix(cli): handle missing language value
refactor(analyzer): separate type checks
docs: update contribution guide
```

### Тип изменения

Для большинства коммитов достаточно пяти основных типов:

| Тип | Когда используется |
| --- | --- |
| `feat` | Новая возможность или новое поддерживаемое поведение. |
| `fix` | Исправление ошибки в существующем поведении. |
| `refactor` | Изменение устройства кода без изменения поведения. |
| `test` | Добавление или исправление только тестов. |
| `docs` | Изменение только документации и примеров. |

`feat` выбирается по изменению поведения, а не по размеру diff. Поддержка новой
синтаксической формы является `feat(parser)`, даже если для неё изменено только
несколько строк. Перенос существующей проверки в отдельный файл без изменения
результата является `refactor(analyzer)`.

### Крупные и специальные случаи

Дополнительные типы используются реже и только тогда, когда весь коммит
действительно относится к указанной задаче:

| Тип | Когда используется |
| --- | --- |
| `perf` | Производительность заметно изменилась и результат подтверждён измерением. |
| `build` | Изменена сборка, версия инструмента, зависимость, Makefile или генератор. |
| `ci` | Изменён только workflow, проверка или окружение CI. |
| `style` | Выполнено крупное механическое форматирование без изменения логики. |
| `chore` | Проведено обслуживание репозитория, которое нельзя точнее назвать другим типом. |
| `revert` | Полностью отменён ранее опубликованный коммит. |

Не используйте `chore` как универсальное название непонятного изменения. Если
коммит исправляет поведение, это `fix`; если добавляет возможность, это `feat`;
если только перестраивает код, это `refactor`.

Примеры специальных случаев:

```text
perf(parser): reduce token path allocations
build(rpl): update generator to v0.7.2
ci: test the host build on pull requests
style(parser): apply project-wide receiver naming
chore: remove obsolete development artifacts
revert: feat(parser): support tuple return types
```

Для `perf` в теле коммита укажите способ измерения и результат до и после:

```text
perf(parser): reduce token path allocations

Reuse the path buffer while parsing qualified types.

BenchmarkParseTypes: 42 allocs/op -> 27 allocs/op
```

Крупное изменение продукта всё равно не оформляется одним общим коммитом
`feat: big update`. Его следует разделить на самостоятельные рабочие части. Если
при этом ломается совместимость, используется `!` и footer `BREAKING CHANGE:`.

### Область изменения

`scope` называет компонент, поведение которого изменилось, а не конкретный файл
или техническую операцию.

Обычные области Candy:

| Область | Содержимое |
| --- | --- |
| `parser` | AST, объявления, выражения, типы и разбор синтаксиса. |
| `lexer` | Токены, литералы, комментарии и позиции. |
| `analyzer` | Анализ AST и проверка типов. |
| `cli` | Команды, параметры и обработка запуска. |
| `composer` | Загрузка файлов и создание проекта. |
| `database` | SQLite, модели RPL и кэш. |
| `errors` | Диагностика и локализованные сообщения. |
| `clifmt` | Форматирование и перевод текста CLI. |
| `lipa` | Визуализация дерева. |
| `build` | Makefile и сборочные сценарии. |

Если изменение относится ко всему репозиторию или область ничего не уточняет,
её можно опустить:

```text
docs: add contribution guide
build: update Go toolchain
```

Не используйте слишком общие области `logic`, `core`, `code`, `misc` и
`changes`. Например, вместо `refactor(logic): update checks` следует написать
`refactor(analyzer): separate type checks`.

### Заголовок

Описание после двоеточия должно:

- начинаться со строчной буквы;
- быть коротким, желательно не длиннее 72 символов вместе с типом и областью;
- описывать один конкретный результат;
- использовать повелительную форму: `add`, `support`, `fix`, `remove`, `split`;
- не заканчиваться точкой;
- не содержать номера файлов, даты и служебные слова вроде `update stuff`.

Хорошие заголовки:

```text
feat(parser): support optional method returns
fix(database): include source text in cache lookup
refactor(cli): extract global option parsing
test(lexer): cover unterminated raw strings
docs: describe RPL generation workflow
```

Неудачные заголовки:

```text
update
fix bugs
parser changes
refactor(logic): ai re
feat: parser
WIP
```

### Тело коммита

Для небольшого очевидного изменения достаточно заголовка. Тело добавляется,
когда без него непонятна причина решения, прежнее поведение или важное
ограничение.

После заголовка оставляется пустая строка. В теле описывается, почему изменение
нужно и как теперь ведёт себя система. Перечислять каждую изменённую строку не
нужно: это уже видно в diff.

```text
fix(parser): preserve block boundary after return errors

Stop return-list recovery at a closing brace so a malformed method does not
consume the following declaration. Keep the partial method in the AST and
report the missing parenthesis at the original boundary.
```

Если решение имеет неочевидное ограничение, оно также указывается в теле:

```text
feat(analyzer): compare complete type paths

Compare token segments directly. Qualified and unqualified names remain
different types until alias resolution is implemented.
```

### Несовместимые изменения

Несовместимое изменение отмечается символом `!` перед двоеточием и поясняется в
footer через `BREAKING CHANGE:`.

```text
refactor(parser)!: store method returns as a slice

Represent zero, one, and multiple return types with a single field.

BREAKING CHANGE: MethodStmt.Return was replaced by MethodStmt.Returns []Type.
```

Одного `!` недостаточно: footer должен объяснять, какой API или синтаксис
изменился и что необходимо исправить пользователю.

### Ссылки на задачи

Ссылки размещаются в footer после пустой строки:

```text
fix(cli): reject missing language values

Closes: #42
```

Используйте `Closes: #42`, если коммит полностью закрывает задачу, и
`Refs: #42`, если он только связан с ней.

### Как разделять большие изменения

Один коммит содержит одно законченное логическое изменение и проходит тесты
самостоятельно. Тесты, необходимые для подтверждения исправления, обычно входят
в тот же коммит, что и код. Не оставляйте промежуточный коммит в состоянии, в
котором проект не собирается.

Большую поддержку возвратов методов можно разделить так:

```text
refactor(parser): store method returns as a slice
feat(parser): parse parenthesized return lists
docs: document method return forms
```

В первом коммите меняется представление AST и обновляются существующие тесты. Во
втором добавляются новый синтаксис, диагностика и тесты этого синтаксиса. Третий
коммит меняет только документацию.

Не следует разделять зависимые части так:

```text
feat(parser): add return parser
test(parser): fix tests after parser change
fix(parser): make previous commit compile
```

Если тесты написаны после реализации, перед отправкой объедините исправляющие
коммиты с соответствующим логическим коммитом.

### Сгенерированные файлы

Изменение схемы RPL и соответствующий сгенерированный код входят в один коммит:

```text
feat(database): cache translations by source text
```

Такой коммит может включать `cache.rpl`, `*.gen.go`, адаптер базы и тесты,
поскольку по отдельности эти части не образуют рабочее изменение. Обновление
версии RPL оформляется отдельным коммитом:

```text
build(rpl): update generator to v0.7.2
```

### Проверка перед коммитом

Название выбирается по staged diff, а не по всему рабочему дереву. Сначала
определите, какие файлы действительно войдут в следующий коммит:

```sh
git status --short
git diff
git diff --cached
git diff --cached --name-only
go test ./...
```

Перед `git commit` проверьте, что в staged diff нет отладочного вывода,
временных файлов и несвязанных правок. Не добавляйте всё через `git add .`, если
рабочее дерево содержит несколько независимых изменений.

Не изменяйте уже опубликованную историю общей ветки. Локальные исправляющие
коммиты можно объединить через `rebase --interactive` до публикации.

### Как выбрать название по текущему diff

Используйте следующий порядок:

1. Посмотрите только staged diff через `git diff --cached`.
2. Одним предложением сформулируйте результат для пользователя или разработчика.
3. Выберите `type` по результату, а не по названиям файлов.
4. Выберите `scope` по компоненту, который владеет изменённым поведением.
5. Добавьте `!`, если изменён существующий API или несовместимый синтаксис.
6. Если одним предложением описать diff нельзя, разделите staged-файлы.

Примеры определения типа:

| Что находится в staged diff | Название |
| --- | --- |
| Парсер начал принимать новую форму метода | `feat(parser)` |
| Исправлено поглощение следующего узла после ошибки | `fix(parser)` |
| Поле AST заменено без нового синтаксиса | `refactor(parser)` |
| Добавлены только тесты существующего поведения | `test(parser)` |
| Переписан только README | `docs` |
| Обновлены RPL и его сгенерированные артефакты | `build(rpl)` |

Если изменение одновременно добавляет новый синтаксис и меняет внутреннее поле
AST, главным результатом является новая возможность. Используйте `feat`, а
несовместимость поля отметьте через `!`:

```text
feat(parser)!: support optional and multiple method returns

Allow methods to omit the return clause or declare a parenthesized list of
return types. Preserve the existing single-return form.

BREAKING CHANGE: MethodStmt.Return was replaced by MethodStmt.Returns []Type.
```

### Случай со смешанным рабочим деревом

Например, одновременно изменены:

```text
internal/parser/stmt.go
internal/parser/parser_test.go
internal/errors/errors.go
README.md
CONTRIBUTING.md
.rpl/attrs/...
```

Название `feat: update project` не подходит: файлы относятся к нескольким
независимым результатам. Разделите их.

Сначала синтаксис, диагностика и его тесты:

```sh
git add internal/parser/stmt.go
git add internal/parser/parser_test.go
git add internal/errors/errors.go
git diff --cached
git commit
```

Сообщение:

```text
feat(parser)!: support optional and multiple method returns

BREAKING CHANGE: MethodStmt.Return was replaced by MethodStmt.Returns []Type.
```

Затем руководство для участников:

```sh
git add CONTRIBUTING.md
git diff --cached
git commit -m "docs: add contribution and commit guidelines"
```

После него основная документация:

```sh
git add README.md
git diff --cached
git commit -m "docs: add Russian project documentation"
```

Обновление RPL выполняется отдельно и только вместе со всеми соответствующими
артефактами:

```text
build(rpl): update generator to v0.7.2
```

Если версия в заголовках `*.gen.go` не совпадает с обновляемой версией RPL,
такой коммит ещё не готов. Сначала перегенерируйте код и проверьте diff.

## Стиль Go-кода

- Перед отправкой запускайте `gofmt` для изменённых Go-файлов.
- Именованный receiver во всём проекте называется `self`.
- Новая абстракция добавляется только тогда, когда она упрощает существующий
  код или соответствует уже используемому шаблону.
- Публичные названия должны объяснять назначение типа или конструктора. Следует
  избегать названий вроде `DB`, `Store`, `New` и `Manager`, если из контекста не
  ясно, что именно они создают или представляют.
- Сгенерированный код вручную не форматируется и не редактируется отдельно от
  генератора.

Пример receiver:

```go
func (self *Parser) parseType() *TypeExpr {
    // ...
}
```

## Изменение парсера

При изменении синтаксиса:

1. Сохраняйте исходные токены и позиции в AST.
2. Не восстанавливайте пути из объединённой строки, если в AST уже есть
   `[]token.Token`.
3. Добавляйте ошибку через `internal/errors`, а не через строку внутри парсера.
4. Добавляйте восстановление до ближайшей безопасной границы конструкции.
5. Проверяйте достижение `EOF`, чтобы повреждённый файл не приводил к
   бесконечному циклу.
6. Добавляйте тесты для корректной формы, ошибочной формы и восстановления.
7. Обновляйте README, если меняется пользовательский синтаксис.

Обычные комментарии `// TODO` об ошибках следует заменять структурированной
диагностикой. Комментарии с пометкой `// TODO NO` являются намеренными и не
изменяются без отдельного решения.

Минимальная проверка парсера:

```sh
go fmt ./internal/parser/... ./internal/errors
go test ./internal/parser ./internal/parser/lexer ./internal/errors
```

## Ошибки и диагностика

Ошибки Candy объявляются в `internal/errors` и выводятся через
[`digreyt`](https://github.com/rp1s/digreyt).

Новая диагностика должна содержать:

- понятное имя кода;
- правильный уровень важности;
- сообщение на английском;
- русский перевод;
- точный диапазон исходного кода;
- подсказку, что ожидалось и как исправить конструкцию.

Предупреждение не должно становиться критической ошибкой. Например, отсутствие
необязательной точки с запятой остаётся `SeverityWarning`.

## RPL и сгенерированная база данных

### Используемая версия

Репозиторий RPL:

<https://github.com/rp1s/rpl>

Candy сейчас использует **RPL `0.7.2`**:

<https://github.com/rp1s/rpl/releases/tag/v0.7.2>

Версия зафиксирована заголовком сгенерированной модели
`internal/database/cache/cache_cli_text/model.gen.go`:

```go
// RPL version: 0.7.2
```

Не используйте более новую версию RPL для обычного изменения схемы. Обновление
RPL выполняется отдельным изменением вместе с проверкой всего сгенерированного
кода.

### Установка RPL 0.7.2

Готовые архивы для macOS, Linux и Windows находятся в выпуске `v0.7.02:

<https://github.com/rp1s/rpl/releases/tag/v0.7.2>

После распаковки держите исполняемый файл `rpl` и каталог `.rpl` из архива
вместе. В `.rpl` находятся генераторы атрибутов и SDK.

Сборка нужной версии из исходного кода:

```sh
git clone --branch v0.7.2 --depth 1 https://github.com/rp1s/rpl.git
cd rpl
make install
```

Проверить установленную версию можно по первой строке справки:

```sh
rpl help
```

Ожидаемое начало вывода:

```text
RPL 0.7.2
```

## Тесты

Основная проверка:

```sh
make test
```

Полная локальная проверка основного модуля:

```sh
go mod tidy
git diff --exit-code go.mod go.sum
go test ./...
make build
```

`pkg/lipa` и `pkg/digreyt` являются отдельными Go-модулями и проверяются
отдельно:

```sh
(cd pkg/lipa && go test ./...)
(cd pkg/digreyt && go test ./...)
```

Если изменение касается только одного компонента, сначала запустите его тесты,
но перед отправкой всё равно проверьте `go test ./...` в корне.

## Документация

README должен описывать только реально поддерживаемое поведение. Не указывайте
генерацию кода, выполнение плагинов или типизацию как завершённые возможности,
пока они не подключены к команде `build`.

Примеры Candy должны разбираться текущим парсером. Для нового синтаксиса
добавляйте пример рядом с соответствующим разделом, не дублируя полное описание
в нескольких местах.

## Перед отправкой изменения

- изменённые Go-файлы отформатированы;
- `go test ./...` проходит;
- независимые модули проверены, если изменение их касается;
- новые ошибки добавлены через `internal/errors`;
- для синтаксиса добавлены положительные и отрицательные тесты;
- сгенерированные файлы соответствуют `model.rpl` и RPL `0.7.2`;
- README обновлён только при изменении пользовательского поведения;
- коммит не содержит несвязанных файлов.
